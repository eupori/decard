## 1. 현황 분석

### 1.1 서비스 개요

데카드는 PDF 학습자료를 업로드하면 AI가 **근거(페이지 번호 + 원문)가 포함된 암기카드**를 자동 생성하는 시험 대비 학습 도구다.

```
핵심 플로우: PDF 업로드 → AI 카드 생성 → 검수(채택/삭제/수정) → 학습(플래시카드)
```

- **프로덕션 URL:** https://decard.eupori.dev
- **기술 스택:** FastAPI(Python) + Flutter(Web/Android) + Claude AI + SQLite
- **타겟:** 대학생, 고등학생, 자격증 준비생

### 1.2 핵심 가치 제안

**"카드 생성이 아니라, 검수 시간을 줄여준다."**

ChatGPT로 카드를 만들 수 있지만, 근거 확인·검수·덱 구조화를 매번 사람이 해야 해서 결국 귀찮아서 안 쓰게 된다. 데카드는 이 전체 과정을 1분 안에 끝낸다.

| 가치 | 설명 |
|------|------|
| **근거 포함 카드** | 모든 카드에 페이지 번호 + 원문 스니펫 자동 첨부. "이 카드가 맞나?"를 5초 안에 판단 |
| **시험형 템플릿** | 정의형("OO란?"), 빈칸형(Cloze), 비교형("A vs B") — 실제 시험 출제 패턴 반영 |
| **검수 워크플로** | 채택/삭제/수정 버튼 중심 UX. 정확도 100%가 아니라 검수 비용 최소화가 목표 |

### 1.3 현재 기능 현황

| 카테고리 | 기능 | 상태 |
|----------|------|------|
| **카드 생성** | PDF 업로드 → AI 카드 자동 생성 (백그라운드) | 완료 |
| **템플릿** | 정의형 / 빈칸형(Cloze) / 비교형 3종 | 완료 |
| **검수** | 채택·삭제·수정·되돌리기 + 전체 채택/해제 | 완료 |
| **학습** | 풀스크린 플래시카드 (스와이프, 셔플) | 완료 |
| **보관함** | 폴더(과목)별 세션 관리, 6색 프리셋, 자동 저장 | 완료 |
| **인증** | 카카오 OAuth 로그인 + 디바이스 ID 폴백 (듀얼 인증) | 완료 |
| **성능** | 청크 병렬 처리(Semaphore=3), JSON 재시도, 프로그레스 UX | 완료 |
| **배포** | Docker Compose + EC2 (Web + API + nginx) | 완료 |
| **플랫폼** | Web + Android APK | 완료 |

### 1.4 경쟁 환경

| 경쟁자 | 강점 | 약점 | 데카드 차별점 |
|--------|------|------|---------------|
| **ChatGPT** | 무료, 즉시 사용 | 덱 구조화·근거·검수 없음, 매번 복붙 필요 | 워크플로 자동화 + 근거 첨부 |
| **Anki** | 무료, 강력한 SRS | 카드를 직접 만들어야 함 (수 시간 소요) | PDF → 카드 자동화 |
| **Quizlet** | 쉬운 UI, 대규모 유저베이스 | 근거 없음, PDF 워크플로 없음, 한국 시장 약함 | 근거 기반 검수 + 한국 시험 최적화 |
| **노트가든** | 유사 컨셉 (PDF→카드) | 서비스 종료 | 검증된 니즈의 시장 빈자리 |
| **족보** | 실제 기출문제 | 과목별 비용, 불법 리스크 | 합법적 개인화 도구 |

**핵심 포지셔닝:** ChatGPT의 카드 생성 능력 + Anki의 학습 구조 + 근거 검수 워크플로를 하나의 제품에 통합.

### 2.2 가격 설정 근거

- **5,900원** = 족보 1과목 구매 가격대(5,000~10,000원)와 일치
- 초기 검증 피드백: "4,900원이면 무조건 쓰죠" → 약간 상향 조정 (5,900원)
- 14일 = 시험 2주 전부터 사용하는 벼락치기 패턴에 최적화
- 학생 가격 민감도: 5,000원 이하 충동 구매, 10,000원 이상 고민 구간

## 5. 퀴즈 모드 설계

### 5.1 퀴즈 유형

#### 객관식 (4지선다)

```
Q: 피아제의 인지발달 이론에서 "감각운동기"에 해당하는 연령대는?

① 2~7세
② 7~11세
③ 출생~2세  ← 정답
④ 11세 이상
```

- AI가 카드 뒷면(정답)을 기반으로 그럴듯한 오답 3개 자동 생성
- 오답 생성 원칙: 같은 카테고리, 비슷한 길이, 흔한 오개념 반영

#### OX 퀴즈

```
Q: "피아제의 감각운동기는 출생부터 약 4세까지이다."

O / X  ← 정답: X (2세까지)
해설: 감각운동기는 출생~약 2세. 2~7세는 전조작기에 해당.
```

- 카드 뒷면을 미묘하게 변형하여 OX 문제 생성
- 오답 시 올바른 정보 + 근거 표시

#### 빈칸 채우기

```
Q: 감각운동기는 출생부터 약 ___세까지로, 영아는 ___을/를 통해 세계를 탐색한다.

정답: 2, 감각과 운동
```

- 기존 빈칸형(Cloze) 카드를 그대로 활용
- 핵심 키워드 자동 마스킹

#### 서술형

```
Q: 피아제의 동화(assimilation)와 조절(accommodation)의 차이를 설명하시오.

[사용자 답변 입력]
→ AI 채점 (기존 grade_service 확장)
→ 평가: 정확도, 핵심 키워드 포함 여부, 피드백
```

- 비교형 카드와 자연스럽게 연결
- AI 채점 기준: 핵심 개념 포함, 논리적 구조, 근거와의 일치도

### 5.2 퀴즈 운영 방식

| 모드 | 설명 | 사용 시점 |
|------|------|-----------|
| **빠른 퀴즈** | 10~20문제, 랜덤 유형 혼합 | 일상 복습 |
| **시험 모의** | 전체 범위, 시간 제한, 채점 | 시험 직전 실전 연습 |
| **오답 집중** | 틀린 문제만 재출제 | 취약 보완 |
| **유형별** | 객관식/OX/빈칸/서술 선택 | 특정 유형 연습 |

### 5.3 AI 기반 오답 생성 전략

```
입력: 카드 앞면 + 뒷면(정답) + 근거 + 같은 세션의 다른 카드들
출력: 그럴듯한 오답 3개

원칙:
1. 같은 범주에서 선택 (날짜면 날짜, 인물이면 인물)
2. 흔한 오개념 반영 (학생들이 실제로 헷갈리는 답)
3. 같은 세션 내 다른 카드의 정답을 오답으로 활용 (혼동 유발)
4. 길이·형식을 정답과 유사하게 (소거법 방지)
```

### 5.4 학습 데이터 분석

| 지표 | 설명 | 활용 |
|------|------|------|
| 문제별 정답률 | 각 카드의 퀴즈 정답률 추적 | 난이도 자동 조정 |

## 6. 기술 아키텍처 로드맵

### 6.1 현재 아키텍처 (Phase 3.5)

```
[Flutter Web/App]
    ↓ HTTP (REST)
[FastAPI + uvicorn]
    ↓ asyncio.create_task
[Claude CLI] ← Semaphore=3 (동시 3개)
    ↓
[SQLite] (단일 파일 DB)
```

**한계:**
- 폴링 기반 진행률 (5~10초 딜레이)
- SQLite 동시 쓰기 제한 (WAL 모드로 완화 중)
- 단일 서버, 수평 확장 불가
- 동시 사용자 ~20명 한계 (테스트 확인: 5명 동시 성공)

### 6.2 Phase 4~5: 실시간 + 안정성

```
[Flutter Web/App]
    ↓ HTTP + SSE (Server-Sent Events)
[FastAPI + uvicorn]
    ↓ asyncio + SSE 스트리밍
[Claude API] ← httpx 비동기 (CLI → API 전환)
    ↓
[PostgreSQL] (동시성 + 안정성)
    ↓
[Redis] (세션 캐시 + 작업 큐)
```

#### SSE (Server-Sent Events) 도입

| 항목 | 현재 (폴링) | 목표 (SSE) |
|------|-------------|------------|
| 진행률 갱신 | 5~10초 딜레이 | 실시간 (~100ms) |
| 서버 부하 | 매 폴링마다 DB 조회 | 이벤트 기반, DB 부하 최소 |
| UX | "진행 중..." 반복 | "3/5 청크 완료", 카드 실시간 추가 |
| 구현 | setInterval + GET | EventSource API + 스트리밍 |

```
SSE 이벤트 형식:
event: progress
data: {"chunk": 3, "total": 5, "cards_so_far": 12, "percent": 60}

event: card_ready
data: {"card": {...}}  // 청크 완료 시 즉시 카드 전달

event: complete
data: {"session_id": "ses_abc123", "total_cards": 20}
```

#### PostgreSQL 마이그레이션

- SQLite → PostgreSQL (동시 쓰기 병목 해소)
- SQLAlchemy ORM 유지 → DB URL만 변경
- Alembic 마이그레이션 도입 (스키마 버전 관리)
- 인덱스 최적화: user_id, folder_id, session_id, created_at

#### Claude CLI → API 전환

- 현재: `subprocess`로 Claude CLI 호출 (프로세스 오버헤드)
- 목표: `anthropic` Python SDK + `httpx` 비동기 호출
- 이점: 프로세스 생성 비용 제거, 스트리밍 응답, 에러 핸들링 개선

### 6.3 Phase 6+: 확장성

```
[CDN] → [Flutter Web SPA]
          ↓
[Load Balancer]
    ↓           ↓
[API Server 1] [API Server 2]  ← K8s Pod
    ↓           ↓
[Redis Queue] (작업 분배)
    ↓
[Worker 1] [Worker 2] [Worker 3]  ← K8s Job
    ↓
[Claude API]
    ↓
[PostgreSQL] (RDS)
```

#### Redis 작업 큐

- 카드 생성 요청을 큐에 등록 → 워커가 순차 처리
- 유료 유저 우선 큐 (priority queue)
- 대기 순번 UI: "앞에 3명 대기 중, 약 2분 후 시작"
- 동시 사용자 100명+ 대응

#### Kubernetes 오케스트레이션

- API 서버: HPA (Horizontal Pod Autoscaler) — CPU/메모리 기반 자동 스케일
- 워커: 큐 길이 기반 스케일 (KEDA)
- DB: AWS RDS PostgreSQL (관리형)
- 스토리지: S3 (PDF 임시 저장, 자동 삭제)

#### CDN

- Flutter 웹 빌드 정적 파일 → CloudFront/Cloudflare CDN
- 글로벌 에지 캐싱 → 초기 로딩 속도 개선
- API는 CDN 미적용 (동적 콘텐츠)

## 7. SRS 반복학습 설계

### 7.1 알고리즘 선택

#### SM-2 → FSRS 전환 계획

| | SM-2 | FSRS |
|---|------|------|
| **개발사** | Piotr Wozniak (1987) | Open-source (2022~) |
| **원리** | 고정 공식, Easiness Factor | 머신러닝 기반, 망각 곡선 모델링 |
| **정확도** | 보통 | 높음 (Anki에서도 채택) |
| **파라미터** | 5개 (고정) | 19개 (학습 데이터로 최적화 가능) |
| **적합성** | MVP 빠른 구현 | 데이터 축적 후 전환 |

**전략:** Phase 5 초기에 SM-2로 시작 → 충분한 학습 데이터 축적 후 FSRS 전환

### 7.2 SM-2 구현 (Phase 5 초기)

```
사용자 자기평가: 0~5점
  0: 완전히 모름 (Blackout)
  1: 틀림, 보고 "아!" (Incorrect, remembered)
  2: 틀림, 하지만 쉽게 떠올림 (Incorrect, easy recall)
  3: 맞힘, 어렵게 (Correct, difficult)
  4: 맞힘, 약간 망설임 (Correct, hesitation)
  5: 완벽하게 맞힘 (Perfect)

복습 간격 계산:
  if grade < 3: interval = 1 (내일 다시)
  elif repetition == 1: interval = 1
  elif repetition == 2: interval = 6
  else: interval = round(prev_interval * easiness_factor)

EF 업데이트:
  EF = EF + (0.1 - (5 - grade) * (0.08 + (5 - grade) * 0.02))
  EF = max(1.3, EF)  // 최소값 제한
```

**UX 단순화:** 5점 척도 → 3단계로 축약

| 버튼 | SM-2 점수 | 의미 |
|------|-----------|------|
| **모르겠다** | 1 | 내일 다시 |
| **애매하다** | 3 | 짧은 간격 복습 |
| **알겠다** | 5 | 긴 간격 복습 |

### 7.3 학습 이력 데이터 모델

```sql
-- 카드별 SRS 상태
CREATE TABLE card_srs_state (
    id TEXT PRIMARY KEY,
    card_id TEXT NOT NULL REFERENCES cards(id),
    user_id TEXT NOT NULL REFERENCES users(id),
    easiness_factor REAL DEFAULT 2.5,
    interval_days INTEGER DEFAULT 0,
    repetitions INTEGER DEFAULT 0,
    next_review_date TIMESTAMP,
    last_review_date TIMESTAMP,
    status TEXT DEFAULT 'new',  -- new, learning, review, relearning
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(card_id, user_id)
);

-- 개별 학습 기록
CREATE TABLE review_logs (
    id TEXT PRIMARY KEY,
    card_id TEXT NOT NULL REFERENCES cards(id),
    user_id TEXT NOT NULL REFERENCES users(id),
    grade INTEGER NOT NULL,        -- 0~5 (SM-2) 또는 1/3/5 (단순화)
    interval_days INTEGER,         -- 해당 시점의 간격
    easiness_factor REAL,          -- 해당 시점의 EF
    review_duration_ms INTEGER,    -- 응답 소요 시간
    reviewed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 7.4 복습 주기 최적화

#### 기본 복습 일정 (SM-2 기반)

```
Day 0: 첫 학습
Day 1: 1차 복습 (모든 카드)
Day 3: 2차 복습 (grade ≥ 3인 카드)
Day 7: 3차 복습
Day 14: 4차 복습
Day 30: 5차 복습 (장기 기억 전환)
```

#### 시험 모드 (Cramming)

- 시험 D-3~D-1: 일반 SRS 무시, 전체 카드 집중 복습
- 취약 카드(EF < 2.0) 우선 출제
- 시험 후: SRS 일정 자동 복구

#### FSRS 전환 시점

- 조건: review_logs 10만 건 이상 축적
- 방법: 기존 학습 데이터로 FSRS 파라미터 최적화
- 기대 효과: 사용자별 망각 곡선 맞춤 → 복습 효율 20~30% 향상 (Anki 벤치마크 기준)

### 7.5 SRS 관련 API

| Method | Path | 설명 |
|--------|------|------|
| GET | `/api/v1/srs/due` | 오늘 복습할 카드 목록 |
| POST | `/api/v1/srs/review` | 학습 결과 기록 + 다음 복습일 계산 |
| GET | `/api/v1/srs/stats` | 학습 통계 (복습 카드 수, 연속 일수, 리텐션) |
| GET | `/api/v1/srs/forecast` | 향후 7일 복습 예정 카드 수 |
